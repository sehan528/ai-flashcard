## 질문 33: Annotation의 동작 원리와 커스텀 Annotation 작성 방법은 무엇인가요?

**정의:**
Annotation은 코드에 메타데이터를 추가하는 선언적 방식으로, 컴파일러나 런타임에 추가 정보를 제공합니다. @ 기호로 시작하며, 클래스, 메서드, 필드 등에 적용되어 코드의 동작을 변경하거나 추가 처리를 지시할 수 있습니다.

**특징/원리:**
- 메타데이터: 코드에 대한 데이터로, 코드 자체는 변경하지 않고 추가 정보만 제공
- 컴파일러 지시: @Override, @SuppressWarnings 등은 컴파일 타임에 검증이나 경고 제어
- 런타임 처리: 리플렉션을 통해 런타임에 어노테이션 정보를 읽어 동적으로 처리
- 타입 안전성: 어노테이션 요소는 타입이 있어 컴파일 타임에 유효성 검증

**주요 메타 어노테이션:**
- @Target: 어노테이션을 적용할 수 있는 위치 지정 (TYPE, METHOD, FIELD 등)
- @Retention: 어노테이션 정보를 유지할 시점 지정 (SOURCE, CLASS, RUNTIME)
- @Documented: JavaDoc에 어노테이션 정보를 포함할지 여부
- @Inherited: 하위 클래스가 상위 클래스의 어노테이션을 상속받을지 여부
- @Repeatable: 같은 어노테이션을 여러 번 적용 가능하도록 지정

**커스텀 Annotation 작성:**
- @interface 키워드로 어노테이션 타입 정의
- 요소는 메서드 형태로 선언하며 기본값 지정 가능
- 요소 타입은 기본 타입, String, Class, enum, 어노테이션, 배열만 가능
- @Retention을 RUNTIME으로 설정해야 리플렉션으로 읽을 수 있음

**어노테이션 처리 방법:**
- 컴파일 타임: Annotation Processor API로 컴파일 시 코드 생성이나 검증
- 런타임: 리플렉션으로 getAnnotation(), isAnnotationPresent() 등을 사용하여 정보 읽기
- 프레임워크 통합: Spring, JPA 등이 어노테이션을 읽어 자동 설정 및 기능 제공

**실무 활용:**
- Spring의 @Component, @Service, @Repository로 빈 자동 등록 및 역할 명시
- JPA의 @Entity, @Table, @Column으로 객체-관계 매핑 정보 선언
- 커스텀 검증 어노테이션을 작성하여 비즈니스 규칙을 선언적으로 표현

## 질문 34: Java의 직렬화(Serialization)와 역직렬화에 대해 설명해주세요.

**정의:**
직렬화는 객체를 바이트 스트림으로 변환하는 과정이며, 역직렬화는 바이트 스트림을 다시 객체로 복원하는 과정입니다. 객체를 파일에 저장하거나 네트워크로 전송하기 위해 사용되며, Serializable 인터페이스를 구현하여 직렬화 가능하게 만듭니다.

**특징/원리:**
- 마커 인터페이스: Serializable은 메서드가 없는 마커 인터페이스로 직렬화 가능함을 표시
- 자동 직렬화: ObjectOutputStream이 객체의 모든 필드를 자동으로 직렬화
- 그래프 순회: 객체가 참조하는 다른 객체들도 재귀적으로 직렬화
- 타입 정보 포함: 클래스 메타데이터가 포함되어 역직렬화 시 타입 복원

**직렬화 프로세스:**
- ObjectOutputStream의 writeObject()로 객체를 바이트 스트림으로 변환
- 모든 인스턴스 필드가 직렬화되며, static과 transient 필드는 제외
- 참조하는 객체들도 모두 Serializable이어야 하며, 아니면 NotSerializableException 발생
- serialVersionUID로 클래스 버전을 식별하여 호환성 관리

**역직렬화 프로세스:**
- ObjectInputStream의 readObject()로 바이트 스트림을 객체로 복원
- 생성자를 호출하지 않고 리플렉션으로 객체를 생성하여 필드 값 설정
- serialVersionUID가 일치하지 않으면 InvalidClassException 발생
- 보안상 신뢰할 수 없는 데이터의 역직렬화는 위험하므로 주의 필요

**커스터마이징:**
- writeObject()와 readObject() 메서드를 정의하여 직렬화 과정 커스터마이징
- writeReplace()로 직렬화 전 다른 객체로 대체
- readResolve()로 역직렬화 후 객체를 변경하거나 싱글톤 보장
- Externalizable 인터페이스로 완전히 수동으로 직렬화 제어

**주의사항:**
- 보안: 역직렬화는 임의의 코드 실행을 유발할 수 있어 신뢰할 수 없는 데이터에는 사용 금지
- 호환성: 클래스 구조 변경 시 이전 버전과의 호환성 깨질 수 있음
- 성능: 직렬화/역직렬화는 느리고 바이트 스트림 크기가 큼
- 대안: JSON, Protocol Buffers 등 더 효율적이고 안전한 직렬화 방식 고려

**실무 활용:**
- 세션 정보를 직렬화하여 분산 환경에서 공유하거나 영속화
- 캐시 저장소에 객체를 직렬화하여 저장하고 필요 시 복원
- 메시징 시스템에서 메시지 객체를 직렬화하여 전송

## 질문 35: serialVersionUID의 역할은 무엇인가요?

**정의:**
serialVersionUID는 직렬화된 클래스의 버전을 식별하는 고유 식별자로, 역직렬화 시 클래스의 호환성을 검증하는 데 사용됩니다. 명시적으로 선언하지 않으면 컴파일러가 자동 생성하지만, 명시적 선언을 권장합니다.

**특징/원리:**
- 버전 관리: 직렬화된 객체와 역직렬화할 클래스의 버전이 동일한지 확인
- 호환성 검증: serialVersionUID가 일치하지 않으면 InvalidClassException 발생
- 자동 생성: 명시하지 않으면 클래스 구조를 기반으로 해시값 자동 생성
- 타입: private static final long으로 선언

**자동 생성의 문제점:**
- 예측 불가능: 컴파일러마다 생성 알고리즘이 다를 수 있어 일관성 부족
- 불필요한 변경: 메서드 추가나 주석 변경 같은 사소한 수정에도 값이 변경될 수 있음
- 호환성 깨짐: 의도치 않게 serialVersionUID가 변경되어 기존 직렬화 데이터를 읽지 못함
- 디버깅 어려움: 호환성 문제 발생 시 원인 파악이 어려움

**명시적 선언 이점:**
- 버전 제어: 개발자가 의도적으로 버전을 관리하여 호환성 결정
- 안정성: 클래스 변경에도 serialVersionUID를 유지하여 하위 호환성 보장
- 명확성: 직렬화 가능 클래스임을 명시적으로 표현
- 성능: 자동 생성 오버헤드 제거

**버전 관리 전략:**
- 초기값: 일반적으로 1L로 시작하여 필요 시 증가
- 호환 가능 변경: 필드 추가, 메서드 추가 등은 serialVersionUID 유지
- 비호환 변경: 필드 타입 변경, 삭제 등은 serialVersionUID 증가
- 마이그레이션: readObject()에서 이전 버전 데이터를 새 버전으로 변환하는 로직 구현

**실무 활용:**
- 모든 Serializable 클래스에 명시적으로 serialVersionUID 선언하여 예측 가능한 동작 보장
- IDE의 경고를 활성화하여 serialVersionUID 누락 시 알림 받도록 설정
- 버전 관리 정책을 문서화하여 팀원들이 일관되게 관리

## 질문 36: Java의 모듈 시스템(Java 9+)에 대해 설명해주세요.

**정의:**
Java 모듈 시스템(JPMS, Java Platform Module System)은 Java 9에서 도입된 기능으로, 코드를 모듈 단위로 구조화하고 명시적으로 의존성을 선언하여 캡슐화와 신뢰성을 향상시킵니다. module-info.java 파일로 모듈을 정의하며, 강력한 캡슐화를 제공합니다.

**특징/원리:**
- 강력한 캡슐화: 명시적으로 export하지 않은 패키지는 외부에서 접근 불가
- 명시적 의존성: requires 키워드로 필요한 모듈을 명시하여 의존 관계 명확화
- 신뢰성: 컴파일 타임과 런타임에 모듈 의존성 검증하여 ClassNotFoundException 방지
- 성능: 필요한 모듈만 로드하여 메모리 사용량 감소 및 시작 시간 단축

**module-info.java 구성 요소:**
- module 선언: 모듈 이름을 고유하게 정의
- requires: 이 모듈이 의존하는 다른 모듈 명시
- exports: 외부에 공개할 패키지 선언
- opens: 리플렉션 접근을 허용할 패키지 지정
- provides와 uses: 서비스 제공자와 소비자 관계 정의

**주요 키워드:**
- requires: 다른 모듈에 대한 의존성 선언, requires transitive로 전이적 의존성 표현
- exports: 패키지를 모든 모듈에 공개, exports to로 특정 모듈에만 공개
- opens: 리플렉션을 위해 패키지를 런타임에 개방, opens to로 특정 모듈에만 개방
- uses: 서비스 인터페이스 사용 선언
- provides with: 서비스 구현체 제공 선언

**모듈 유형:**
- 명명된 모듈: module-info.java를 가진 모듈로 강력한 캡슐화 적용
- 자동 모듈: module-info가 없는 JAR를 모듈 경로에 두면 자동으로 모듈화되며 모든 패키지 export
- 무명 모듈: 클래스 경로에 있는 모든 코드로 모듈 시스템 밖에 존재

**실무 활용:**
- 대규모 프로젝트를 모듈로 분리하여 명확한 경계와 의존성 관리
- 내부 API를 숨기고 공개 API만 export하여 캡슐화 강화
- 서비스 로더 패턴을 uses와 provides로 구현하여 플러그인 아키텍처 구축
